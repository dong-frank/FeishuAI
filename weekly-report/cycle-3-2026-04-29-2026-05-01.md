第三周期（4.29-5.1）
一、核心产出 （必填）
本周期最重要的产出，是把 git-helper 从“功能 demo”继续推进成了一个可以被稳定验证的实验系统。第二周期已经有了 Git Agent 和 Lark Agent 的协作主链路，但当时最大的问题是：这个工具到底在什么真实场景中有价值、怎么复现、怎么判断 Agent 的回答是不是有效。这个周期主要围绕这个问题，搭建了一个可复现实验场景，假想了在一个scrum 团队中的新人开发者完成一天的需求开发的故事，把 commit message、冲突处理、upstream 报错、push 后协作通知等场景串成一条可重复演示的链路。在复赛提交demo的时候，通过这一个需求场景的演示，配合具体评估数据来证明价值。

另一个比较满意的产出，是对 Git Agent 本身做了更接近真实使用的打磨：加入了跨阶段记忆、上下文压缩、token / context usage 统计、输出校验和主动对话入口。这样 git-helper 不再只是“按 Tab 或命令失败后被动触发 Agent”，而是开始支持用户在 TUI 中直接向 Agent 追问，或者指示完成更加复杂的任务。

三、过程复盘与沉淀（必填）
1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？
这周最核心的是搭建 FlowDesk 实验。之前一直在做工具能力，但缺少一个稳定的业务验证样本，所以这次设计了一个虚拟 Scrum 团队：Scrum Master、Product Owner、Senior Developer 和 New Developer。初步设计了以下场景故事：New Developer 饶东申领取 FD-124“工单列表支持按优先级筛选”，站会结束后回到终端，用 git-helper 完成开发过程中的 Git 操作。这个故事把工具要证明的价值具体化了：新人不熟悉团队 Git 规范时怎么生成 commit message，遇到 merge 冲突时怎么按团队排障手册处理，没有 upstream 时怎么给出正确 push 命令，push 后又如何进入协作 review 流程。

为了让这个场景可重复运行，本周期补了 experiments/flowdesk 目录，包括 scenario.yaml、项目 fixture、本地 Markdown 飞书文档替身、预期用例，以及 reset / export / score 脚本。这样每次可以通过 npm run experiment:flowdesk -- reset --stage ... 把 demo 项目恢复到指定状态，不用手工造冲突、造 staged diff 或造 upstream 报错。这个对后续录屏、答辩、评估 Agent 输出都有帮助。

Git Agent 侧这周改为了和Lark Agent类似的行为，具备上下文短期记忆，来让Git Agent给出更自然的补全帮助。同时又做了 history compaction，只把用户的任务，和最终的返回放进历史上下文，避免把 stdout、stderr、git status、tuiSession 这类通过工具实时获取的上下文塞进历史，导致后续回答被旧状态污染和上下文膨胀。

在交互体验上，本周期补了 /chat。原本 Command Agent 只能通过三阶段触发：按 Tab 的 beforeRun、命令成功后的 afterSuccess、命令失败后的 afterFail。现在在 TUI 输入 /chat 怎么解决 upstream 报错，会直接把消息交给 Command Agent，不走 shell 执行，也不会记录 Git 命令统计。这个入口让用户可以主动追问，也让 demo 中“先问清楚再操作”的场景更自然。也为后续Lark Agent 进行一些敏感操作，如给联系人发送消息，留下交互接口。

2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？
最大的不顺还是怎么设计我的验证场景。如果只说 git-helper 可以帮助 Git 新手，听起来比较泛；如果只展示 git status、git push这种命令帮助，容易显得 Agent 价值不高，因为很多开发者本来就会用 Git。后来把场景收敛到实际企业中的一次新人（可能是实习生）入组：新人不仅是不懂 Git 命令，还不熟悉团队规范、分支命名、commit message、冲突处理、需求状态流转和 review 协作。这些内容正好需要“通用 Git 知识 + 团队飞书知识库”的组合，能够体现工具和飞书集成的意义。在下一周期中还需要进一步精化这个场景设计，展示处理复杂问题的能力。

另一个技术卡点在模型调用链路上。为了让 TUI 能针对模型回复做结构化解析和定制渲染，我们在模型调用时引入了结构化输出。但豆包 2.0 当时还不支持 OpenAI 兼容接口里基于 JSON Schema 的结构化输出，只能通过 function calling 来约束返回格式。
最初我使用了 LangChain 内置的隐藏结构化输出工具，但很快遇到一个诡异问题：模型调用看起来成功了，LangSmith 里也能看到 token usage，最终传到 TUI 的回复内容却经常为空。起初怀疑是上下文过长导致，但后来发现即使在会话刚开始、上下文很短时也会触发，于是开始怀疑是豆包 2.0 对这套结构化输出机制的兼容性问题。
为了定位问题，又切到 Qwen3 做对比测试，结果问题反而更明显：业务工具调用已经完成后，模型会反复调用 LangChain 隐藏的结构化提取工具，持续输出类似内容，直到达到工具调用次数上限。这个现象基本确认了问题不在业务逻辑本身，而是在“最终结构化回复”的结束机制上。
后来通过和 AI 一起分析，我们放弃依赖 LangChain 隐藏的结构化输出工具，改成显式注册一个 final_response工具，由模型在完成业务工具调用后主动调用它返回最终结构化结果。这个工具设置为直接返回，从而明确告诉 LangChain 本轮调用已经结束。这样既保留了 TUI 所需的结构化输出，又避免了隐藏工具导致的空回复和重复调用问题。

四、本周随手记（选填）
这个周期最大的感受是，做 Agent 项目不能只堆功能，必须尽早设计“如何证明它有用”。之前做完 Git Agent 和 Lark Agent 后，功能上是兴奋的，但也会担心它是不是只是在简单 Git 命令上套了一层很重的 AI。实验场景搭起来后，这个问题稍微清楚一些：工具的价值不一定在于解释 git push 本身，而在于把用户当前命令、团队规范、需求上下文和协作动作串起来。

另外也更明显感受到，CLI Agent 的体验很受延迟影响。能用系统补全解决的，就不要交给 Agent；能用确定性 runtime 分支解决的，也不要让模型推理。Agent 更适合处理上下文整合、规范解释、错误诊断和协作建议。
