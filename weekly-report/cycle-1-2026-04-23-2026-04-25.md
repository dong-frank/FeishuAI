三、过程复盘与沉淀（必填）
1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？
这周主要先把产品形态从“命令行工具”重新收敛成“带 TUI 的 Git 辅助终端”。TUI 层使用 Ink 和 React 实现，支持用户输入命令、展示历史输出、识别退出命令，并加入了 Git 命令输入过程中的 Tab 补全能力。这个部分让 git-helper 不再只是 git-helper run xxx 这种一次性命令，而是开始有一个持续交互的工作台。

runtime 层主要完成了命令解析、命令分类和执行流程的拆分。当前命令会被分成支持的 Git 命令、自定义命令和其他命令；命令末尾加 ? 会被识别成帮助请求，进入 askForHelp流程，并且不会真正执行原命令。这个设计为后面“执行前提示”“执行失败后诊断”“执行成功后总结”等能力留出了稳定入口。

Agent 层从 OpenAI Agents SDK 的尝试，调整为 LangChain 的通用 agent 封装。模型配置支持 baseURL、api key、模型名，并默认关闭 thinking；后面又按 LangChain 官方 quickstart 改成 createAgent({ model, tools, systemPrompt }) 的方式，让工具调用由 LangChain 自己编排。当前已经定义了 COMMAND_AGENT_TOOLS，并接入第一个工具 tldr_git_manual，用于在 git status ?、git push ? 这类帮助场景下查询通用 Git 手册。

飞书相关目前还停留在基础 wrapper 和产品设计阶段，没有真正接入飞书 API 的业务闭环。已经做了 lark-cli 的状态、setup、login、搜索文档等包装，也讨论过未来如何让 Agent 查询飞书知识库、选择组织 Git 规范文档、以及在 PR 场景中通知维护者，但这一周期没有把飞书 API 集成进核心运行链路。还需要考虑到的一个问题是飞书文档上如果同样的内容过于分散，如何高效寻找，在目前我的设计中，我会希望用户来指定一个团队的Git 规范文档，后续该设计是否合适还要进一步思考。

AI 分工上，主要让 AI 帮忙拆产品结构、识别边界、写 TypeScript 骨架、补测试、查 LangChain 文档并根据实际报错修正实现。我主要负责判断产品方向，例如从纯 CLI 调整成 TUI，从“马上接飞书”调整为“先打通 runtime + agent + tool”，以及确认哪些能力先不接入。

2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？
一个比较明显的卡点是 Agent 工具调用的实现方式。一开始只是把 tools 通过 model.bindTools(tools) 绑定到模型上，但这并不等于工具会被真正执行。后来查了 LangChain 官方 quickstart，发现现在推荐直接使用 createAgent，由 LangChain 封装工具调用循环，于是把自定义 pipe 链路改成了官方 agent 形态，并补了 fake model 的测试来确认 tool call 确实会触发工具执行。

另一个卡点是 prompt 模板中的 {} 问题。之前 system prompt 里放了类似 JSON 的示例，LangChain 的 ChatPromptTemplate 会把单个 `{` 当成模板变量解析，导致 Single '}' in template 报错。后面通过移除 prompt 中的原始 JSON 示例，改成字段列表描述，绕开了模板解析问题。这个坑也提醒后续写 system prompt 时，如果要展示 JSON，要么转义花括号，要么避免使用会被模板引擎解析的形式。

还有一个产品层面的卡点是飞书接入范围容易变大。飞书知识库检索、文档选择、登录、搜索、PR 通知都很有价值，但如果同时推进，很容易把 CLI 主流程拖散。最后的破局方式是先承认飞书 API 这一周期不接入核心流程，只保留 wrapper 和设计入口，先把 TUI、runtime、agent、tool 这些主骨架做扎实。

3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？
最可复用的是 runtime 的 phase 设计。askForHelp、beforeRun、afterSuccess、afterFail 这几个阶段把 Agent 介入命令生命周期的位置说清楚了，后面无论接 tldr、飞书知识库、Git 报错诊断还是 PR 通知，都可以挂在这些阶段上，不需要重新设计主流程。

第二个可复用的是工具数组模式。COMMAND_AGENT_TOOLS 现在已经是一个清晰入口，后续可以继续加入 search_lark_docs、read_lark_doc、notify_reviewer 等工具。Agent 不需要知道每个工具背后的实现细节，只需要通过工具描述和 schema 判断什么时候调用。

第三个可复用的是“通用知识 + 团队知识”的检索思路。tldr 适合解决 Git 命令基础用法，飞书知识库适合解决团队规范和内部流程，Agent 负责把两者合并成简短、可执行的建议。这个分层非常适合继续扩展到报错诊断场景。

这次还沉淀了一个项目内 skill：weekly-cycle-report。之后可以让AI先帮助生成一份周期记录并保存到 weekly-report/ 目录下，然后我再对内容进行调整，减少每次重新组织周报结构的成本。
