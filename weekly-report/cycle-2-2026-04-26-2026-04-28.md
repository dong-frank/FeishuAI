一、核心产出 （必填）
本周期最重要的产出，是把第一周期的idea做成了实际可用的demo，推进成一个具备明确 Git Agent 和 Lark Agent 分工协作的版本。第一周期主要证明了 TUI、runtime、Command Agent 和基础工具可以跑起来；第二周期则重点补上了“什么场景由哪个 Agent 处理、什么时候查询飞书、什么时候只给本地建议、结果如何回到终端”的边界。

当前V1.0 demo版本中主要覆盖以下功能场景
1. 基础命令的实时帮助推送：用户在CLI输入任意字符后，按Tab都可以触发Git Agent的command help 功能，当前核心围绕Git这一场景设计。Git Agent 会先使用tldr去查阅手册，在手册基础上，直接给出建议命令和相应的解释说明。
[图片]
2. 基于企业知识库的知识内容推送：在1功能基础上，对于需要查询企业知识库的帮助，或执行失败的命令，如生成commit 的message，Git Agent 会和Lark Agent 交互，由Lark Agent去查阅对应的飞书文档并返回显示给用户，同时补充建议命令的完整性或给出对应的排障步骤。
[图片]
3. 命令执行成功后在飞书进行对应消息的推送：当部分命令执行成功后，Git Agent 和 Lark Agent 交互，推送更新部分消息，如完成git push 后给仓库的维护者发送消息，或者是更新对应的需求文档，这部分具体的场景还待进一步细化。

三、过程复盘与沉淀（必填）
1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？

这周主要定下了系统中两个主要Agent的协作方式，我区分了一个Git Agent 和 一个 Lark Agent。因为对于 Git Agent来说，不是每次触发它都需要调用飞书的API，和飞书进行交互，而且在需要飞书交互的过程中，会产生大量的飞书专有的上下文，如用户的登录授权信息，让一个专门的Lark Agent来管理所有需要和飞书交互的操作，在系统设计上比较合理。

同时也实现了Lark Agent和接入飞书API，在周期一中因为担心接入范围过大遗留的问题也得到解决，采取给Agent暴露自由执行lark-cli 的工具，但严格受控执行的方法。针对我目前需要的场景，参考官方的skill，编写我自己的lark skill，如lark-doc-lookup，就是一个专门用来查飞书文档的skill。Git Agent在需要飞书交互的时候，会下方具体受控的任务，lark agent只能参照对应任务的skill进行操作，有效避免了一下接入太多功能导致混乱的情况发生。当前的skill只覆盖了基本的业务场景，后续再进行扩展和完善。

2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

也就是关于Agent的协作方式，最开始设想的是采用Orchestrator调度的方法，让它来带调度Git Agent 和 Lark Agent的执行，并完成最终的汇总展示，但问题也很明显，多一个Agent就会多出好几轮多推理时延。在当前我设想的这个CLI场景的Agent中，用户操作输入一条命令的生命周期是相对确定的，可以整体划分为 before-run、after- success、after-fail。根据这个生命周期来直接调度Git Agent的相关行为是比较自然的。

但第二个问题就是，如何与飞书进行交互，比如在 before-run 阶段，查询帮助时需要企业知识库的内容，需要查阅飞书文档。如果直接把相关的lark-cli 作为工具暴露给Git Agent，就会遇到前文涉及的那个问题，把飞书专有的上下文大量的塞给Git Agent，Git Agent除了针对命令的处理，还要考虑是否完成登录授权等信息。最终确定再独立一个Lark Agent，并把整个和Lark Agent 的交互封装为工具暴露给Git Agent。在Git Agent侧，只需要在需要飞书交互的时候，给出对应的任务描述，由Lark Agent完成具体的交互任务，再把有用的内容返回，既避免了多轮的Agent调度，也能有效控制各自的能力边界。

3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？

把具体的业务场景封装为Skill的方法，在之后的实践中还可以继续复用。因为在项目初期阶段，设想的场景需求还相对比较简单，且会频繁迭代。在这次项目中，我就先把涉及到的几个场景单独封装，如command-before-run，刚开始这个skill只涉及查阅 tldr 手册，再接入Lark Agent 后，直接在这个skill中补充，复杂问题下去查阅飞书文档。而且在当前我这个系统中，由于有相对确定的生命周期，可以在运行时动态注入skill，也不用让Agent自己去找了。

四、本周随手记（选填）
在实际写代码前习惯了先和AI交流讨论意见，就比如使用codex的plan模式，我先给它描述一个大概的方向和自己的思路，AI会来向你确认具体路线，给出完整的计划。这个过程中不只是让AI更加清楚接下来要做什么，我觉得更是让自己的逻辑更加清晰。

同时随着项目的推进，有了一个可以实际使用的demo后，我反而又开始担心最初的选题问题。担心在真实企业场景中并不存在这样的需求。客观来说，git 这一场景很简单，是开发人员每天都会接触到的命令，相对来说已经比较熟悉，再引入agent，能提供多少有价值的帮助？而且带来的推理延时和tokens消耗，在这样简单的场景中是否有必要？不过当初选择这个场景，也是因为自己相对熟悉，同时身边的同学们，很少会去用git，甚至有研究生学长说嫌麻烦没用过，对于刚离开校园的职场新人来说，有这样一个工具应该也会很有帮助。

在后面的时间中，需要着重考虑这个验证问题，关键在于构想设计出一个好的验证场景，在场景中实际使用这个项目，来证实其价值。

在本周开发我的Lark Agent 的时候，还发现了一个小问题，就是用lark-cli 我可以去搜索查看组织内其他人开启互联网分享权限的文档，就比如这次比赛的个人阶段成功小结。没有lark-cli之前，我们似乎无法访问到别人开启权限，但没有主动把链接分享给你的内容。对于部分的技术大牛，他们可能会去翻阅飞书的API手册，然后用API查询的方式去获取。但现在每个人的AI就是一个技术大牛，我们给它配备好lark-cli后，它就能做任何事，甚至给我们一些设想之外的内容，可能会引发一系列的问题。
